13 /*
14  * XPLM Data Access API - Theory of Operation
15  *
16  * The data access API gives you a generic, flexible, high performance way to
17  * read and write data to and from X-Plane and other plug-ins. For example,
18  * this API allows you to read and set the nav radios, get the plane location,
19  * determine the current effective graphics frame rate, etc.
20  *
21  * The data access APIs are the way that you read and write data from the sim
22  * as well as other plugins.
23  *
24  * The API works using opaque data references. A data reference is a source of
25  * data; you do not know where it comes from, but once you have it you can
26  * read the data quickly and possibly write it. To get a data reference, you
27  * look it up.
28  *
29  * Data references are identified by verbose string names
30  * (sim/cockpit/radios/nav1_freq_hz). The actual numeric value of the data
31  * reference is implementation defined and is likely to change each time the
32  * simulator is run (or the plugin that provides the datareference is
33  * reloaded).
34  *
35  * The task of looking up a data reference is relatively expensive; look up
36  * your data references once based on verbose strings, and save the opaque
37  * data reference value for the duration of your plugin's operation. Reading
38  * and writing data references is relatively fast (the cost is equivalent to
39  * two function calls through function pointers).
40  *
41  * This allows data access to be high performance, while leaving in
42  * abstraction; since data references are opaque and are searched for, the
43  * underlying data access system can be rebuilt.
44  *
45  * A note on typing: you must know the correct data type to read and write.
46  * APIs are provided for reading and writing data in a number of ways. You can
47  * also double check the data type for a data ref. Note that automatic
48  * conversion is not done for you.
49  *
50  * A note for plugins sharing data with other plugins: the load order of
51  * plugins is not guaranteed. To make sure that every plugin publishing data
52  * has published their data references before other plugins try to subscribe,
53  * publish your data references in your start routine but resolve them the
54  * first time your 'enable' routine is called, or the first time they are
55  * needed in code.
56  *
57  * X-Plane publishes well over 1000 datarefs; a complete list may be found in
58  * the reference section of the SDK online documentation (from the SDK home
59  * page, choose Documentation).
60  *
61  */

606 /***************************************************************************
607  * SHARING DATA BETWEEN MULTIPLE PLUGINS
608  ***************************************************************************/
609 /*
610  * The data reference registration APIs from the previous section allow a
611  * plugin to publish data in a one-owner manner; the plugin that publishes the
612  * data reference owns the real memory that the data ref uses. This is
613  * satisfactory for most cases, but there are also cases where plugnis need to
614  * share actual data.
615  *
616  * With a shared data reference, no one plugin owns the actual memory for the
617  * data reference; the plugin SDK allocates that for you. When the first
618  * plugin asks to 'share' the data, the memory is allocated. When the data is
619  * changed, every plugin that is sharing the data is notified.
620  *
621  * Shared data references differ from the 'owned' data references from the
622  * previous section in a few ways:
623  *
624  * - With shared data references, any plugin can create the data reference;
625  * with owned plugins one plugin must create the data reference and others
626  * subscribe. (This can be a problem if you don't know which set of plugins
627  * will be present).
628  *
629  * - With shared data references, every plugin that is sharing the data is
630  * notified when the data is changed. With owned data references, only the one
631  * owner is notified when the data is changed.
632  *
633  * - With shared data references, you cannot access the physical memory of the
634  * data reference; you must use the XPLMGet... and XPLMSet... APIs. With an
635  * owned data reference, the one owning data reference can manipulate the data
636  * reference's memory in any way it sees fit.
637  *
638  * Shared data references solve two problems: if you need to have a data
639  * reference used by several plugins but do not know which plugins will be
640  * installed, or if all plugins sharing data need to be notified when that
641  * data is changed, use shared data references.
642  *
643  */
644
